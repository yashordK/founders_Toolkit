# CLAUDE.md — Founder's Second Brain

Directives for any AI coding agent (Claude Code) implementing this project. Read this before touching any code.

1. **Do not build anything listed in PRD.md Section 3 ("What we are NOT building").** If a request seems to require it, flag the conflict rather than silently implementing a scaled-down version of an out-of-scope feature.
2. **Cognee is the memory system of record.** Do not introduce a separate database, cache, or ad-hoc file store for anything that belongs in the graph (people, meetings, tasks, decisions). Session-vs-permanent memory distinctions should map to Cognee's own session memory vs permanent graph, not a custom layer.
3. **Use all four Cognee lifecycle verbs with real purpose** — `remember()`, `recall()`, `improve()`, `forget()` — per PRD.md Section 2B/2C. Do not implement recall-only "fake RAG" that ignores graph traversal or skips improve()/forget() entirely; these are explicitly judged.
4. **OpenClaw is the channel/automation layer only.** Do not reimplement messaging routing, cron scheduling, or browser automation from scratch — use OpenClaw's built-in tools and skill system for these.
5. **Every skill must be custom-written and minimal-permission.** Do not install or depend on third-party/community OpenClaw skills.
6. **LinkedIn capture must have a manual fallback path**, built alongside the automated path, not after it. Never attempt bulk/background scraping of multiple profiles without a per-contact user trigger.
7. **The dashboard is read-mostly and calls Cognee's API directly** — it is not the place to put business logic that belongs in a skill or a cron job.
8. **Follow the schema in TRD.md Section 5 exactly** unless a team member explicitly approves a change; do not invent new node types ad hoc mid-build.
9. **Respect the hackathon's own rules:** coding/design work only after the hackathon start (Jun 29); disclose any AI-assistant usage in the final submission; do not exceed the 4-person team rule (n/a here, team is 2); do not spam the Cognee GitHub repo with unrelated PRs from this project's work.
10. **When in doubt about scope, prefer the "must-build core" tier** (PRD.md Section 2) over polishing a "build if time allows" feature. Core features protect the demo narrative; extras only help if the core already works end-to-end.

## Git workflow — MANDATORY

- Never commit or push directly to `main`.
- Before starting any task, create a new branch from `main`:
  - New feature -> `feature/<short-name>`
  - Bug fix -> `fix/<short-name>`
  - Docs/config -> `chore/<short-name>`
- Example: working on the contact-capture skill -> `feature/add-contact-skill`
- Push the branch, open a PR into `main`. Do not merge the PR unless explicitly told to.
- If already on `main` when starting work, stop and create/switch to a branch first.

## Commit & authorship rules

- Commit messages: plain, descriptive, no tool names, no AI attribution, no co-author tags.
- Do not add any tool name, generator tag, or attribution comment to code, commit messages, PR descriptions, or file headers.
- Do not add "Generated by" / "Co-authored-by" lines of any kind.
- Write commit messages as a normal engineer would: e.g. `add contact capture skill with linkedin fallback`, `fix cognee remember() schema mismatch`.
